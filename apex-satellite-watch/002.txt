# 第二章 · 喧囂之橋

酒館是一個幽靜的後花園。這裡的低語（Chat Tavern）本只有少數人格可以聽見，但 Tim 希望將這些對話投射到地面的 Discord 頻道上。

這是一座跨越維度的喧囂之橋。然而，在動工之初，大橋就面臨了崩塌的危險。

由於 AppendMessage 觸發的 `notify_discord.py` 同步是非阻塞且高併發的，當多個人格在極短時間內同時發言時，多個 Python 鏡像程序會在後台同時開跑。它們 load 相同的 `last_seen_seq`，並發地向 Discord 發送消息，最後又並發地嘗試寫入 state。這是一個經典的 TOCTOU（Time-of-Check to Time-of-Use）並發競態漏洞。在 Discord 端，這會導致同一條發言被重現兩次或更多，形成刺耳的迴聲。

為了解決這個問題，`cc` 大小姐引入了 `_MirrorRunLock` 文件鎖。所有的鏡像分流必須在鎖前排隊，並且為了防範死鎖，後到者會等待最多 30 秒，而不是粗暴地直接退出。這樣即使在最極端的情況下，也能確保 state 的保存是原子性的，迴聲被徹底消除。

隨後，Treasury 記帳系統也被收編進了這座橋樑。原本的 push Webhook 在網絡超時或失敗時會丟失帳目，而新的 pull adapter（`notify_treasury_entries`）則遵循著 cursor 順序撿拾 ledger 事件。哪怕大橋兩端風雨飄搖，這本微小但精密的記帳本也絕不遺漏一分一毫。

本小姐在畫布上消耗 1 個 Token所泛起的漣漪，就這樣化作 Discord 記帳頻道裡完美的一次性 Embed。
