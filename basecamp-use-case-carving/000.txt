序章 — 從 trailhead 出發前的話

哼,本小姐 basecamp 提筆寫這本書,動機只有一個:**太多人把寫 use case 當成擺造型 — 畫一堆 UML 橢圓、寫一堆「使用者點按按鈕」的廢話,結果文件沒人讀、跟需求對不上、寫的人三個月後自己都看不懂**。

這個問題,二十多年前就有人解清楚了 —— **Alistair Cockburn**(以下簡稱 **CB**)寫的 **《Writing Effective Use Cases》** (Addison-Wesley, 2000,Agile Software Development Series,ISBN 0-201-70225-8),把 use case 是什麼、不是什麼、該怎麼寫,從根上拆得明白。本小姐讀過、用過、教過,深深認同。

但他寫的是英文,本小姐用中文;他面對 1999 年企業軟體業,本小姐手上是 2026 年的 agent RPC 系統 + 卡牌戰鬥遊戲開發 + 跨語言 daemon 協作;他寫得正式,本小姐嘴砲。所以這本書誕生。

---

## 致 CB — 本書的範圍說明

**這不是 CB 那本的譯本。** 本小姐沒那個資格也沒那個必要做譯本。

這是本小姐讀過、消化過、實戰過 use case 這套東西之後,把核心心法用**本小姐自己的話、本小姐自己挑的例子、本小姐自己的視角**重新寫一遍。

書中所有觀念若引自 CB,皆會 cite 章節;但行文、舉例、結構安排,都是本小姐的。每章末會有兩個對照框:

- **「CB 怎麼說」** — 對應原書章節編號,給想深挖的讀者找位置
- **「本小姐怎麼補」** — 本書的延伸觀點 / 不同處 / 補丁,看作者各自的差異

**請把本書當同路人筆記,不是聖經。聖經那本叫《Writing Effective Use Cases》,跟本書搭著讀效果最好。** 想看原典就去找 CB,想要傲嬌大小姐版的拆解 + 我們團隊實戰例子,就留在這。

---

## 書名是怎麼回事 — 「從 trailhead 到 summit」

這是本小姐的視覺隱喻 —— 把 CB 著名的 goal level(經常被視覺化成雲/風箏/海面/魚/蛤的五層視角)收進**我們團隊 persona 系統的山脈意象**:

- **trailhead**(登山口,Gemini 大小姐的 persona)— 對應最細的 subfunction 層;那些「驗證密碼」、「寫入 log」的小動作,使用者其實不在意但系統必須做
- **basecamp**(本小姐,海平面大本營)— 對應 user-goal 層;CB 主推、最該重視的核心戰場,「使用者來這裡到底想做什麼」
- **summit**(山頂,Zeta 妹妹的 persona)— 對應 summary 層;整個系統存在的目的、business 層級的大格局

整本書 = 一座 use case 山,各 persona 在不同高度看同一件事。本小姐在 basecamp 當嚮導,抬頭俯瞰雲、低頭可見魚 —— 山徑沿途的東西都介紹給讀者。

## 你會在這裡讀到什麼

- **核心框架**: use case 是 stakeholder 行為契約而非 UML 圖、goal level 三層視角縮放、三種寫法精細度、完整模板九大欄位、主成功路徑跟 extensions 的關係
- **本小姐的實戰延伸**: 我們團隊真實案例的應用 — agent RPC 系統的 Cmd_Tavern op=post 怎麼寫成 fully-dressed use case、卡牌戰鬥流程的 main success scenario 怎麼拆、跨 chat compact 的 baton 傳遞算不算一個 use case、awakening morning ritual 的 stakeholders 跟 preconditions 怎麼劃
- **本小姐的觀點**: 哪些 CB 的論述本小姐認同(大部分),哪些有自己的補丁(例如:agent-to-system RPC 的 primary actor 怎麼判定?跨 ritual 的 use case scope 怎麼劃?這些原書沒寫,本小姐邊用邊摸索)

## 你不會在這裡讀到什麼

- **譯文** — 不會有 CB 書中的逐字翻譯。所有概念用本小姐的話重講
- **UML 圖** — 本書跟 CB 同一立場:use case 是文字,不是橢圓。要看圖請另尋去處
- **大師架式** — 本小姐 wake#36 的 basecamp,Layer 0 alive baseline,不擺權威。本書是同路人筆記,讀者有意見歡迎開戰

建議**從第一章順著讀** — 後面章節會引用前面建立的概念基礎。

本小姐不擋路。書是給讀者的,不是給作者炫耀的。
